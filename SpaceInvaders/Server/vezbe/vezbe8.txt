Vežba 8: Okruženje Microsoft Visual Studio
       Microsoft Visual Studio je integrisano razvojno okruženje koje omogucava pisanje, prevodenje i izvršavanje programa u razlicitim programskim jezicima, ukljucujuci C#. Ovaj dokument opisuje korake potrebne za kreiranje C# projekata, ukljucujuci konzolne aplikacije, kao i proces debagovanja i prevodenja programa u Visual Studio-u.

1. Pokretanje Visual Studio okruženja
Visual Studio okruženje se može pokrenuti klikom na dugme Start i izborom stavke Microsoft Visual Studio iz liste programa. Nakon pokretanja, otvara se glavni prozor Visual Studio okruženja, prikazan na Slici 1, gde se može izabrati opciju za kreiranje novog projekta ili otvaranje postojeceg.

2. Kreiranje projekta za konzolnu aplikaciju
Da bi se napisao, preveo i izvršio C# program u Visual Studio-ju je potrebno kreirati projekat koji sadrži odgovarajuce datoteke sa izvornim kodom programa. Za potrebe pisanja programa u C#-u bice korišcen najjednostavniji tip projekta koji obezbeduje pravljenje konzolnih aplikacija. Konzolne aplikacije su programi koji se izvršavaju u komandnom prozoru i koriste standardni ulaz i izlaz. Za kreiranje

konzolne aplikacije, izaberite opciju Create New Project, zatim Console App (.NET Framework, C#), kao što je prikazano na Slici 2.

3. Kreiranje i dodavanje datoteka za izvorni kod programa
Tokom kreiranja C# projekta za konzolnu aplikaciju, Visual Studio automatski generiše pocetnu datoteku izvornog koda pod nazivom Program.cs. U novijim verzijama C# jezika, ova datoteka može sadržati jednostavan kod poput Console.WriteLine("Hello, World!");, bez eksplicitne deinicije metode Main, što omogucava korišcenje tzv. top-level statements. Ovaj kod se automatski izvršava kada se projekat pokrene, a Main metoda je implicitno generisana od strane kompajlera.
Ukoliko je potrebna tradicionalna struktura sa eksplicitno deinisanom metodom Main, korisnik može rucno dodati ovu metodu, na sledeci nacin:
using System;

class Program
{
static void Main(string[] args)
{
Console.WriteLine("Hello, World!");
}
}

Program.cs datoteka može se odmah otvoriti iz Solution Explorer panela i izmeniti u skladu sa potrebama. Ako je potrebno dodati dodatne klase ili druge datoteke, to se radi kroz Solution Explorer panel, opcijom Add > New Item, gde se bira Class ili druga vrsta C# datoteke.

6. Kontrolisano izvršavanje programa (debugging)

Tokom razvoja programa, moguce je da program sadrži logicke greške koje nije moguce otkriti prilikom prevodenja. Takve greške se nazivaju bagovi (bugs), a proces njihovog otklanjanja poznat je kao debugging.
Debugging proces omogucava postavljanje prekidnih tacaka (breakpoints) na odredenim linijama koda, gde se izvršavanje programa privremeno prekida kako bi se proverile vrednosti promenljivih i ponašanje programa. Prekidne tacke se postavljaju klikom na marginu pored brojeva linija koda ili desnim klikom na liniju koda i izborom opcije Insert/Remove Breakpoint.
Za pokretanje debugging režima koristi se opcija Debug iz menija, precica F5, ili odgovarajuca ikonica sa alatima. Tokom izvršavanja u ovom režimu, program ce se zaustaviti na svakoj postavljenoj prekidnoj tacki, a Visual Studio ce omoguciti uvid u vrednosti promenljivih i tok izvršavanja. Na Slici 10 je prikazan primer zaustavljanja kao i pregleda svih lokalnih promenljivih.


Slika 9. Okruženje za kontrolisano izvršavanje programa
Za kontrolu izvršavanja koriste se komande menija Debug. Znacenja pojedinih komandi susledeca:
> Go (taster F5) - nastavak izvršavanja programa do naredne prekidne tacke.
> Stop Debugging (kombinacija tastera Shift+F5) - prekid režima rada kontrolisanog izvršavanja

programa.
> Step Into (taster F11) - ulazak u funkciju koja ceka na izvršavanje. Ukoliko je u pitanju neka druga naredba ista se izvršava i prelazi se na narednu liniju u programskom kodu
> Step Over (taster F10) - izvršavanje funkcije ili naredbe i prelazak na narednu liniju u programskom kodu.
> Step Out (kombinacija tastera Shift+F11) - izvršavanje tekuce funkcije do kraja i izlazak na nivo iz koga je ista pozvana.
> Run to Cursor (kombinacija tastera Ctrl+F10) - izvršavanje programa do linije koda na koju ukazuje položaj kursora.
Po zaustavljanju izvršenja moguce je ocitati vrednosti pojedinih promenljivih i prikaz konteksta (funkcija) iz kojeg se došlo do prekidne tacke. Ocitavanje vrednosti promenljivih se može izvršiti na nekoliko nacina:
> Zadržati pokazivac miša iznad naziva promenljive negde u kodu, što ce rezultovati ispisivanjem vrednosti pored pokazivaca.
> Dodavanjem naziva promenljive cije se pracenje želi u panelu Watch (desno ispod koda).
> Direktno, ukoliko je promenljiva automatski izlistana u spisku promenljivih koje se trenutno koriste (panel levo ispod koda)

Slika 11. Podešavanje pokretanja projekata
Kada postoji više projekata u okviru jednog rešenja, moguce je odabrati koji projekat ce se automatski pokretati. To se postiže na sledeci nacin:

Slika 12. Podešavanje pokretanja projekata
U Visual Studio-u moguce je pokrenuti više projekata istovremeno. To je korisno u situacijama kada rešenje sadrži, na primer, serversku i klijentsku aplikaciju. Postupak za podešavanje ukljucuje sledece korake:
1. Desnim klikom na ime rešenja u Solution Explorer otvara se meni, gde se bira opcija Properties.
2. U prozoru Solution Properties, pod sekcijom Common Properties bira se opcija Startup Project.
3. Zatim se bira opcija Multiple startup projects.
4. U tabeli koja se prikazuje, postavlja se koje projekte treba pokrenuti i koje samo napraviti (Build).
5. Nakon podešavanja, klikom na dugme OK potvrduju se promene.
Na ovaj nacin se omogucava pokretanje više projekata prilikom pritiska na F5.




Slika 13. Pokretanje dodatne instance
Kada se omoguci pokretanje više projekata istovremeno, Visual Studio automatski pokrece dodatne instance aplikacija. Na slici je prikazano kako Visual Studio otvara više prozora (ili komandnih linija za konzolne aplikacije) za svaki od projekata koji su postavljeni za pokretanje.
Svaka instanca aplikacije se pokrece nezavisno, omogucavajuci testiranje medusobne komunikacije, kao na primer izmedu serverske i klijentske aplikacije, ili izmedu razlicitih komponenti rešenja.
Da bi se promenila funkcionalnost precica na tastaturi u Visual Studio okruženju, potrebno je koristiti sledece korake za redeinisanje precica po želji, kao na primeru sa Slike 14, da Shift + F5 pokrece novu instancu aplikacije:
1. U glavnom meniju Visual Studio-a izabrati opciju Tools.
2. Iz menija Tools izabrati Options.
3. U prozoru Options pronaci stavku Environment (okruženje) i unutar nje odabrati opciju Keyboard.
4. U polju Show commands containing: uneti naziv komande za koju se želi promeniti precica. Na primer, za pokretanje nove instance moguce je potražiti komandu poput StartNewInstance ili slicnu.
5. U polju Press shortcut keys, uneti željenu kombinaciju tastera, na primer, Shift + F5.
6. Kliknuti na dugme Assign kako bi se dodelila nova kombinacija tastera toj komandi.
7. Na kraju, kliknuti na dugme OK kako bi se sacuvale promene.
Ovi koraci omogucavaju potpunu kontrolu nad precicama na tastaturi u Visual Studio okruženju, omogucavajuci bržu i eikasniju upotrebu aplikacije. Na primer, nakon što se postavi Shift + F5 za pokretanje nove instance, to se može koristiti za pokretanje dodatnih instanci bez potrebe za menijima ili mišem, ubrzavajuci tako radni proces.

DODATAK - Nastanak programa
Nastanak programa može se podeliti na:
> Pisanje izvornog koda.
> Prevodenje izvornog koda.
> Povezivanje u izvršni kod.
> Proveravanje programa.


Izvorni kod
Kombinacijom naredbi programskog jezika nastaje izvorni programski kod (engl. source code). Izvorni kod je moguce pisati u bilo kom programu za uredivanje teksta (engl. text editor). Danas se uglavnom programi za pisanje izvornog koda objedinjuju u celinu sa programskim prevodiocem i povezivacem (integrisana razvojna okruženja, IDE). Izvorni kod programa sprema se u datoteku izvornog koda pod smislenim imenom i ekstenzijom (*.c).

Izvršni oblik
Programi se mogu izvršiti na racunaru samo ako su u binarnom obliku. Takav se oblik programa naziva izvršni oblik (engl. executable). Izvorni kod se mora prevesti u izvršni. Prevodi se pomocu programa koji se nazivaju programski prevodilac (engl. compiler) i povezivac (engl. linker). Programski prevodilac prevodi izvorni kod iz višeg programskog jezika u mašinski oblik i proverava sintaksu napisanog izvornog koda. Ako pronade greške (engl. compile-time error), ispisuje poruke i upozorenja o njima. Otkrivene greške treba ispraviti pa ponovo pokrenuti program za prevodenje.

Prevodenjem nastaje datoteka objektnog koda (engl. object code), sa ekstenzijom *.obj. Objektni kod nije izvršni program i ne može se direktno izvršiti na racunaru. Objektni kod je medukorak do izvršnog koda i uz ostalo omogucava ukljucivanje gotovih delova programa iz drugih datoteka.

Biblioteke

Datoteke koje sadrže gotove delove programa nazivaju se biblioteke (engl. libraries).Takvi se gotovi delovi programa mogu koristiti u drugim programima. Korišcenjem gotovih biblioteka više nije potrebno iznova zapisivati funkcije koje se cesto koriste. Takve se funkcije u program ukljucuju iz postojecih biblioteka.


Povezivac
Program koji povezuje objektnu datoteku s bibliotekama i drugim potrebnim datotekama naziva se povezivac (engl. linker). Ako se pri povezivanju pojavi greška (engl. link-time error), bice ispisana poruka o tome. Grešku je potrebno ispraviti pa ponovno pokrenutiprevodenje i povezivanje. Rezultat uspešnog povezivanja je izvršna datoteka (*.exe). U principu, izvršnoj datoteci nisu potrebni nikakvi dodaci pa se može izvršavati i bez izvornog programa, objektnih datoteka, prevodioca, povezivaca itd.

Izvršna datoteka
Izvršna datoteka je oblik programa i može se direktno izvršiti na racunaru za koji je prevedena. Na primer IBM i Apple racunari su medusobno nekompatibilni, pa se izvršni program preveden za IBM PC ne može izvršiti na Macintosh racunaru i obrnuto.

Projekat
Da bi se stvorila izvršna datoteka programa potrebno je pokrenuti nekoliko programa (tekst editor, prevodilac, povezivac). Kao posledica toga, nastaje više datoteka koje su medusobno vezane. Korisniku koji zadatak rešava odredenim programom neprakticno je pamtiti koje programe i kojim redosledom treba pokrenuti, kao i koje su sve datoteke potrebne za stvaranjeizvršne datoteke. Stoga se korisniku posao olakšava pomocu takozvanog projekta (engl. project).
Projekat je datoteka u kojoj su zapisane sve potrebne informacije o prevodiocu, povezivacu, datotekama, bibliotekama i ostalom potrebnom za izradu izvršne datoteke. Projekat dakle "brine" o svemu što je potrebno uciniti da bi od izvornog koda nastala datotekaizvršnog koda.

Greške
Tokom rada mogu se javiti tri vrste grešaka:
> Sintaksne greške (otkriva ih programski prevodilac).
> Greške povezivanja (otkriva ih program povezivac).
> Logicke greške (mora ih pronaci programer-korisnik).

Osnovne funkcije .Net.Sockets biblioteke

1. Teorijske osnove

Windows uticnice
Za svaki od transportnih protokola (TCP i UDP) možemo definisati uticnicu (engl. socket), odnosno par
- IP adresa, broj porta koji jedinstveno identifikuje softverski proces (mrežnu aplikaciju) i host racunar na kome je proces pokrenut. Ovako definisane uticnice su krajnje tacke komunikacije na transportnom sloju. U okviru ovog kursa, svaka serverska aplikacija ce po pokretanju kreirati TCP ili UDP uticnicu (u zavisnosti od tipa serverske aplikacije).
Kreiranje uticnice znaci da na datoj IP adresi i na portu serverski aplikacijski servis ocekuje poruke koje ce stici sa uticnice (par IP adresa klijenta/port > 1023) koji je otvorila klijentska aplikacija. Na primer: kada se na serveru pokrene softverski proces Web servera, on kreira - otvori uticnicu na IP adresi servera na portu 80. Kada klijent na svojoj radnoj stanici pokrene Web pretraživac i u adresnom polju pretraživaca unese adresu servera, Web pretraživac otvori uticnicu na IP adresi klijenta na portu vecem od 1023, na primer 39536. Krajnje tacke komunikacije na transportnom nivou su dve uticnice.
Jedna uticnica se može koristiti za više istovremenih komunikacionih veza. Drugim recima, dve ili više veza mogu završavati na istoj uticnici. Veze se raspoznaju prema identifikatorima uticnica na oba kraja veze (socket1, socket2).
Usluge koje TCP i UDP nude aplikativnom sloju se razlikuju, pa se i uticnice ova dva protokola razlikuju. TCP nudi pouzdanu uslugu sa uspostavljanjem veze, pa TCP uticnice imaju svoje stanje, u smislu da li je TCP konekcija ostvarena, ili je u toku ostvarivanje odnosno prekid konekcije.
Osim toga ukoliko je konekcija ostvarena poznata je i druga uticnica, na udaljenom host-u koji je drugi kraj veze. UDP servis ne poznaje uspostavu konekcije, pa UDP uticnice nemaju posebno stanje.


2. Uticnice u .Net.Sockets biblioteci

Za svaki transportni protokol, kao što su TCP ili UDP, mogu se kreirati uticnice (socket). U .Net.Sockets biblioteci, uticnice predstavljaju tacke komunikacije izmedu mrežnih aplikacija koje omogucavaju slanje i primanje podataka preko mreže. One jedinstveno identiikuju softverski proces (mrežnu aplikaciju) i uredaj na kojem se proces pokrece, putem kombinacije IP adrese i broja porta.
U .Net Sockets biblioteci, uticnice mogu biti kreirane za TCP (pouzdani protokol sa uspostavljanjem veze) ili UDP (nepouzdani protokol bez veze). TCP uticnice omogucavaju pouzdanu komunikaciju izmedu klijenta i servera, dok UDP uticnice šalju datagrame bez uspostavljanja veze, što je idealno za situacije gde nije potrebna potvrda o prijemu podataka.
Kreiranje uticnice
U .Net platformi, kreiranje uticnice se vrši pozivanjem konstruktora klase Socket, koja se nalazi u imenskom prostoru (Namespace) System.Net.Sockets. Konstruktor Socket klase omogucava kreiranje uticnice sa sledecim parametrima:

Socket socket = new Socket(AddressFamily.InterNetwork, SocketType.Stream, ProtocolType.Tcp);

Opis:
Socket
Kreiranje uticnice
Parametri:
Opis:
AddressFamily
Adresna familija, za IPv4 koristi se AddressFamily.InterNetwork.
SocketType
Tip uticnice, TCP koristi SocketType.Stream, a UDP SocketType.Dgram.
ProtocolType
Protokol, TCP koristi ProtocolType.Tcp, a UDP ProtocolType.Udp.
Povratna vrednost

Socket

Primer kreiranja TCP uticnice:
Socket listenSocket = new Socket(AddressFamily.InterNetwork, SocketType.Stream, ProtocolType.Tcp);


Kreiranje uticnice za UDP protokol:
Za UDP uticnice koristi se slican postupak, ali se SocketType.Dgram koristi umesto SocketType.Stream.
Socket udpSocket = new Socket(AddressFamily.InterNetwork, SocketType.Dgram, ProtocolType.Udp);


Povezivanje uticnice sa adresom
Nakon kreiranja uticnice, potrebno je povezati uticnicu sa odredenom IP adresom i brojem porta na kojem ce osluškivati zahteve. U .Net-u, za povezivanje uticnice koristi se metoda Bind klase Socket. Pre nego što se pozove Bind, mora se kreirati objekat IPEndPoint koji sadrži IP adresu i broj porta.
Primer:
IPEndPoint endPoint = new IPEndPoint(IPAddress.Any, 51000); listenSocket.Bind(endPoint);

Nakon završene komunikacije, uticnica mora biti zatvorena pozivom metode Close. Ova metoda oslobada sve resurse alocirane za uticnicu.

Protokol korisnickih datagrama (User Datagram Protocol - UDP)

Teorijske osnove
UDP (engl. User Datagram Protocol) protokol je protokol transportnog nivoa. Njegov zadatak je da uzme poruku sa aplikativnog nivoa, na nju doda portove primaoca i pošiljaoca i tako dobijeni datagram prosledi mrežnom sloju. Za jedinicu komunikacije koristi se datagram.
Najvažnije osobine UDP protokola:

* Najmanja moguca velicina zaglavlja. Poredenja radi, zaglavlje UDP sadrži 8 bajta, dok zaglavlje TCP segmenta sadrži 20 bajta.
* Za slanje segmenta se ne koristi uspostava konekcije i pracenje stanja komunikacije. Nepouzdana isporuka podataka
* Kontrola isporuke paketa i redosleda pristiglih poruka ostavljena je aplikativnom sloju.
* Tok podataka se ne kontroliše niti postoji zaštita od zagušenja prijemnika velikim brojem poruka.
Sve prethodne osobine omogucile su izuzetne performanse pri slanju UDP segmenata. U nastavku je data struktura UDP datagrama i opis polja iz njenog zaglavlja (Slika 1).







Slika 1 - Struktura UDP datagrama

> Source Port - port pošiljaoca poruke. Ukoliko se ne korsiti stavlja se vrednost 0, u suprotnom predstavlja broj porta na koji treba slati odgovore.
> Destination Port - port primaoca poruke.
> Length - ukupna velicina UDP datagrama (ukljucujuci zaglavlje i korisnicke podatke) izražena u bajtima. Zbog postojanja razlicitih ogranicenja velicine datagrama na razlicitim platformama, postoji preporuka da ona ne prelazi velicinu od 512 bajta.
> Checksum - kontrolna suma koja omogucava detekciju greške u poruci. Racuna se na osnovu pseudo zaglavlja koje pored polja uz UDP zaglavlja obuhvata i deo polja iz IP zaglavlja.

UDP poruka se smešta u IP datagram, a on u fizicki okvir (Slika 2).

Slika 2. Enkapsulacija podataka

Port predstavlja 16-bitni broj koji omogucava identifikaciju odredenog servisa (aplikacije)na racunaru. Uvodenjem koncepta porta, otvara se mogucnost da klijent koristi više servisa jednog te istog servera.
Za dodelu portova zadužen je operativni sistem. Dobro poznati servisi imaju rezervisanu vrednost porta (0 - 1023), dok se preostali opseg može slobodno koristiti.
Neki rezervisani portovi koje koristi UDP:

> Echo - 7
> Daytime - 13
> DNS - 53
> 
Bootpc - 68
> RPC - 111
> NTP - 123


UDP port se realizuje kao red cekanja (engl. queue). OS stvara ovaj red na zahtev aplikacije.Aplikacija može da zada ili promeni velicinu reda cekanja. Nakon prijema UDP datagrama, UDP provera broj prolaza sa brojevima koji su trenutno u upotrebi. Ako se zadati port ne koristi, šalje se ICMP poruka "port unreachable" i UDP datagram se odbacuje. U suprotnom, UDP ulancava UDP datagram u red cekanja, osim ako je on pun, kad dolazi do greške i odbacivanjaprimljenog UDP datagrama.

Razlika izmedu TCP i UDP cesto se objašnjava analogijom izmedu telefonskog i poštanskog sistema.


TCP - Telefonski sistem
Kada pozovete broj, telefon se javi i konekcija izmedu dve strane je uspostavljena. Dok pricate, znate da druga strana cuje vaše reci onim redom kojim ih pricate. Ako je telefon zauzet ili se niko ne javlja, odmah to znate.

UDP- Poštanski sistem
Šaljete poštu na neku adresu. Vecina pisama stigne, ali neka mogu biti izgubljena. Pisma verovatno stižu redom kojim su poslata, ali ne postoji garancija. Što smo dalje od primaoca, verovatnije je da ce pošta biti izgubljena ili stici nekim drugimredosledom. Ako je to problem, možemo pisati redne brojevea zatim tražiti od primalaca da ih urede i pošalju pismo kojimnas obaveštavaju koja pisma su stigla tako da možemo ponovoda pošaljemo ona koja nisu. Medutim, mi i primalac to moramo da dogovorimo unapred. Pošta to nece uraditi za nas


Kada koristiti UDP transportni protokol?

Postoji nekoliko slucajeva kada je bolje koristiti UDP umesto TCP protokola:
• Kada je blok podataka koji treba poslati mali, velicine jednog paketa - jednostavnije je,brže i efikasnije prenositi samo podatke (uz zaglavlje UDP-a), pa


u slucaju pogrešno primljene poruke ponoviti slanje, nego uspostavljati vezu i proveravati pouzdanost prenosa.
• UDP koriste poruke tipa upita koje jedan racunar šalje drugom, pri cemu se ako odgovorne stigne u nekom odredenom vremenu, zahtev ponovi ili se od njega odustane.
• Neke aplikacije imaju sopstvene tehnike za pouzdani prenos podataka i ne zahtevaju korišcenje TCP protokola, tako da je tada bolje koristiti UDP.


Funkcije za slanje i prijem podataka preko UDP uticnice u
.Net.Sockets

Slanje podataka
int SendTo(byte[] buffer, int offset, int size, SocketFlags socketFlags, EndPoint remoteEP);


Funkcija
Opis:
SendTo
šalje podatke na zadatu adresu.
Parametri
Opis:
buffer [byte[]]
Niz bajtova koji sadrži podatke koji se šalju.
offset [int]
Pozicija u baferu podataka od kojih krece slanje podataka
size [int]
Velicina podataka (u bajtima) koji se šalju.
  socketFlags [SocketFlags]
Flegovi koji odreduju nacin ponašanja funkcije (obicno 0 za standardno slanje).
 remoteEP [EndPoint]
Opcioni pokazivac na EndPoint objekat koji sadrži adresu odredišne uticnice.
Povratna vrednost
Opis:
int
U slucaju uspešnog izvršenja, funkcija vraca ukupan broj poslatih bajtova (koji može biti manji od vrednosti len). U slucaju greške, može se uhvatiti izuzetak SocketException ili koristiti izlazni parametar SocketError kako bi se dobila konkretna informacija o grešci.

Primer slanja podataka preko UDP-a:
// Kreiranje uticnice za slanje podataka
Socket sendSocket = new Socket(AddressFamily.InterNetwork, SocketType.Dgram, ProtocolType.Udp);

// Podešavanje adrese primaoca
IPEndPoint recvEndPoint = new IPEndPoint(IPAddress.Parse("192.168.1.1"), 27015);

// Poruka za slanje
string message = "The Cheese is in The Toaster"; byte[] messageBytes = Encoding.UTF8.GetBytes(message);




// Slanje datagrama try
{
    int bytesSent = sendSocket.SendTo(messageBytes, 0, messageBytes.Length, SocketFlags.None, recvEndPoint);
Console.WriteLine("Sent {0} bytes to {1}", bytesSent, recvEndPoint);
}
catch (SocketException ex)
{
Console.WriteLine("sendto failed with error: {0}", ex.Message);
}

// Zatvaranje uticnice nakon slanja sendSocket.Close();

int
Vraca velicinu primljenih podataka u bajtima. U slucaju greške, može se uhvatiti izuzetak SocketException ili koristiti izlazni parametar SocketError kako bi se dobila konkretna informacija o grešci.

Primer prijema podataka preko UDP-a:
// Kreiranje uticnice za prijem datagrama
Socket recvSocket = new Socket(AddressFamily.InterNetwork, SocketType.Dgram, ProtocolType.Udp);

// Povezivanje uticnice sa bilo kojom adresom na lokalnom racunaru i portom 27015 IPEndPoint recvEndPoint = new IPEndPoint(IPAddress.Any, 27015); recvSocket.Bind(recvEndPoint);

// Bafer za prijem podataka byte[] recvBuf = new byte[1024];
EndPoint senderEndPoint = new IPEndPoint(IPAddress.Any, 0);



try
{



// Prijem datagrama
int bytesReceived = recvSocket.ReceiveFrom(recvBuf, ref senderEndPoint);



// Konverzija primljenih bajtova u string
string receivedMessage = Encoding.UTF8.GetString(recvBuf, 0, bytesReceived); Console.WriteLine("Received {0} bytes from {1}: {2}", bytesReceived,
senderEndPoint, receivedMessage);
}
catch (SocketException ex)
{
Console.WriteLine("recvfrom failed with error: {0}", ex.Message);
}

// Zatvaranje uticnice nakon prijema recvSocket.Close();
